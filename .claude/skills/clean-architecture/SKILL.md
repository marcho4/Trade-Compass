# Описание архитектуры 

Архитектура сервиса состоит из пяти слоев с направлением зависимостей внутрь — внешние слои зависят от внутренних, но никогда наоборот.

- adapter - здесь обрабатываются входящие запросы (http, grpc, message broker). Принимает данные в собственных DTO (request/response), маппит их в доменные сущности и вызывает usecase. Зависит от: usecase, domain.

- domain — ядро приложения. Содержит сущности, value objects и бизнес-правила. Не зависит ни от одного другого слоя и не содержит DTO или транспортных структур.

- gateway - клиенты для взаимодействия с внешними системами (HTTP, gRPC, message broker). Имеет собственные DTO для сериализации/десериализации внешних контрактов и маппит их в доменные сущности. Реализует интерфейсы, объявленные в usecase. Зависит от: usecase (через интерфейсы), domain.

- repository - работа с хранилищами данных (БД, кеш). Имеет собственные модели (например, ORM-структуры) и маппит их в доменные сущности. Реализует интерфейсы, объявленные в usecase. Зависит от: usecase (через интерфейсы), domain.

- usecase - бизнес-логика фич. Оперирует доменными сущностями, объявляет интерфейсы для repository и gateway, не зная об их конкретных реализациях (Dependency Inversion). Зависит от: domain.

Правило зависимостей: adapter → usecase → domain ← repository, gateway. Интерфейсы объявляются там, где используются — в usecase. Слои repository и gateway реализуют эти интерфейсы, а конкретные реализации инжектируются через adapter или DI-контейнер. Прямые зависимости между adapter, gateway и repository запрещены.

Правило DTO: каждый внешний слой (adapter, gateway, repository) владеет собственными DTO/моделями для работы с транспортом и хранилищем. Маппинг в доменные сущности происходит на границе слоя. Доменные сущности никогда не содержат тегов сериализации (json, xml, gorm и т.д.).

## Важные замечания

1. Обработка ошибок между слоями. Это одна из самых часто обсуждаемых тем. Ключевая идея: инфраструктурные ошибки не должны бесконтрольно «протекать» в доменный слой — это антипаттерн. Нужно зафиксировать: domain определяет свои типы ошибок (бизнес-ошибки), repository/gateway оборачивают инфраструктурные ошибки на границе слоя, adapter маппит доменные ошибки в HTTP-коды / gRPC-статусы. domain определяет var ErrOrderNotFound = errors.New(...), repository возвращает эту ошибку вместо sql.ErrNoRows, adapter маппит её в 404

2. Тестирование по слоям. Каждый слой тестируется независимо: зависимости мокаются через интерфейсы. Для usecase мокаются repository и gateway. Для adapter мокается usecase. Интеграционные тесты выносятся отдельно, так как им нужна инфраструктура.

3. Usecase не вызывают другие usecase. Usecase'ы не должны взаимодействовать друг с другом. Если есть общая логика — она выносится в доменный сервис или в отдельный shared-сервис внутри domain.

4. Где живёт DI / wiring (точка сборки). Infrastructure-пакеты содержат конкретные клиенты (Postgres, Redis, AMQP), логгер, конфигурацию HTTP-сервера. Нужно явно указать, где происходит «сборка» приложения — main / cmd / composition root, чтобы Claude не инжектил зависимости внутри usecase. `internal/app/app.go` точка сбора должна быть тут.

5. Контекст (context.Context). Нужно прокидывают ctx через все слои.

6. Логгер — пользуемся Slog. В начале функции можно создать локального логгера через slog.With() и накидать в него параметров, которые зафиксированы в функции для контекста. Логгер - это slog.Default(). (т.е достаточно вызвать slog.Info и тд)

7. Статический анализ архитектуры. Можно использовать линтер go-cleanarch, чтобы автоматически проверять правильность зависимостей между слоями. Для скилла это полезно как правило ревью.

## Антипаттерны

**Нарушения зависимостей между слоями:**
1. usecase импортирует пакет adapter, repository или gateway напрямую
2. domain импортирует любой другой слой
3. repository импортирует adapter или gateway
4. gateway импортирует adapter или repository
5. Прямая зависимость между adapter и repository/gateway (минуя usecase)

**Нарушения domain:**
6. Доменные структуры содержат теги сериализации (json, xml, gorm, db, bson)
7. Domain импортирует инфраструктурные пакеты (database/sql, net/http, конкретные ORM)
8. Бизнес-правила валидации размазаны по adapter или usecase вместо domain

**Нарушения usecase:**
9. Usecase вызывает другой usecase
10. Usecase содержит инфраструктурный код (прямые SQL-запросы, HTTP-вызовы, работа с файлами)
11. Usecase создаёт конкретные реализации зависимостей вместо получения через DI
12. Интерфейсы для repository/gateway объявлены не в usecase, а в самих реализациях
13. Usecase возвращает DTO адаптера или модели репозитория вместо доменных сущностей

**Нарушения DTO и маппинга:**
14. Доменные сущности используются как request/response в adapter (протечка домена в транспорт)
15. ORM-модели из repository передаются напрямую в usecase без маппинга
16. DTO gateway (внешний контракт) просачивается в usecase

**Нарушения обработки ошибок:**
17. Инфраструктурные ошибки (sql.ErrNoRows, http.StatusNotFound) протекают в usecase или domain
18. adapter возвращает доменные ошибки клиенту без маппинга в HTTP-коды / gRPC-статусы
19. Ошибки проглатываются без логирования и оборачивания (голый `_ = err`)
20. `fmt.Errorf` без `%w` — потеря цепочки ошибок

**Нарушения context и логирования:**
24. Функция принимает или передаёт данные без `context.Context` первым аргументом
25. Логгер создаётся через `log.Println` или `fmt.Println` вместо `slog`
26. В логах отсутствует контекст операции (request_id, user_id, entity_id)

27. Если в domain появляются сущности Response / Request - ошибка в архитектуре. Это должны быть DTO, которые мапятся в entity из домена
